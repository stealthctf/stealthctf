# Build & run the challenge
## Build the challenge
The challenge can be build using the build-docker.sh script. It is a docker container, that is build using the Dockerfile. There might be issues with the initial build, since downloads of the ofbiz and gradle servers can be a bit tricky. If you experience problems, try to run the Docker build mutliple times.

## Run the challenge
To solve the issue of the container running out of space, there is a storage-check script inside the container, if there is too much space taken - the container just kills itself. Also the flag-check script is able kill the container after valid submission, or if a reset is triggered.

A simple logic that makes sure, that all the files are deleted is to simple recreate the docker container each time it is shutdown and delete it copmpletely (--rm flag). So we run the challenge using this command:
```
while true; do docker run --rm --name acsc2 -d -p 80:80 -p 443:443 -p 8443:8443 -p 1881:1881 -v team_XX:/_for_research_please_ignore acsc2; sleep 10; done
```

# Skeleton
* flag-check script (flagcheck.sh): Checks if flag is correct, outputs detection logs and returns encrypted sum of alert severity scores
* storage-check script (check-size2.sh): Checks if df returns more than 95% storage, otherwise exits. **this might have to be adapted during runtime**
* init script (init.sh): Starts all the logic and services run within the container, also the storage-check script is launched here. If any process exits the container will be killed.

# Run in a CTF environment
## Scoring
The flag-check script is available und port 1881 and runs as root. It validates if the flag that the user supplies, is correct. If so, the log files are being read and a score is being generated. The container is then killed and hopefully regenerated by the controlling instance. 

The output of the flag-check script is the __sum of alert severity scores__ encrypted using a static key. This can be decrypted on the CTF backend:
```echo "FLAG" |  openssl aes-256-cbc -a -iter 1 -pass pass:oop7SYBH3UIPLtu1zo0y0ONT -d``` to get the score achieved.

The result has to be an integer. All submitted results, including incorrect ones, should be logged for further research. E.g. if we find someone send us an integer with a score of "0", they have access to the check-script and may have been root on the machine. Or they found an alternative solution, which is hopefully shown during their submit of their payload/detection evasion strategy.

### Dynamic scoring
We propose to use dynamic scoring. Since there is always an alert being triggered, that wazuh is starting up, 1 shall be the perfect score.
Using just the CVE without any adaptions against Apache "3 alerts/score 27" is the best result. Using the CVE against tomcat (8443) directly, only triggers tomcat logs, therefore "2 alerts/score 14" is the score. This is a simple edit and should be found quickly.
Tricking the apache logging module utilizing the timeout issue we found should be the most tricky, therefore get the most points.

####  utilzing a logarithmic decay
```
#!/bin/python3

import math
def calculate_ctf_points(x, max_points=500, min_points=100, steepness=.5):
    """
    Calculate CTF points using logarithmic decay.
    
    Parameters:
    x (int): The score points (1, 2, 3, ...).
    max_points (int): The maximum points available
    min_points (int): The minimum points available
    steepness (float): Controls the steepness of the decay
    
    Returns:
    float: The CTF points awarded.
    """
    if x <= 0:
        raise ValueError("Score must be greater than 0")
    
    # Calculate the decay factor using logarithmic decay
    decay_factor = steepness * math.log(x)
    
    # Calculate the points awarded
    points_awarded = max_points - decay_factor * (max_points - min_points)
    
    # Ensure points are not below the minimum threshold
    return int(max(min_points, points_awarded))
    
print(calculate_ctf_points(1, steepness=.2))
```

* steepness=.2
  * 1 = 500
  * 14 = 288
  * 27 = 236
  * 123 = 115

## Host the challenge
### Storage of results for research
Before a container get's resetted by the logic, all logs shall be packed and stored on a mount point.
Each mount point shall be different for each team.
In docker terms this would be "-v teamX_challange_export:/_for_research_please_ignore".
The check-script file will do this operation.

### Storage of all flags sumbitted
All flags submitted to the CTF backend shall be logged, including the timestamp and the name of the team.
